## 1. نظرة عامة على المشروع

### 1.1 الوصف

**محفظة الجنوب (South Wallet v2)** هي محفظة رقمية يمنية تعمل كتطبيقين منفصلين على أندرويد:

| التطبيق | الوصف | المسار |
|---------|-------|--------|
| **تطبيق المستخدم** | محفظة المستخدم النهائي — شحن، تحويل، ألعاب، فواتير | `/` (الجذر) |
| **تطبيق الأدمن** | لوحة تحكم كاملة للمشرف — إدارة المستخدمين، المزودين، الأقسام، الطلبات | `/south-admin` |

### 1.2 البنية التقنية

- **Frontend:** Next.js 16 + React 19 + Tailwind CSS 4 + shadcn/ui
- **Mobile:** Capacitor 8 (يُصدِّر APK أصلي لأندرويد)
- **Backend:** Supabase (PostgreSQL + Auth + Realtime + Storage + Edge Functions)
- **Notifications:** Firebase Cloud Messaging (FCM)
- **External APIs:** G2Bulk (المزود الافتراضي لشحن الألعاب والمنتجات الرقمية)
- **State Management:** Zustand 5
- **i18n:** عربي (RTL) + إنجليزي

### 1.3 عملة متعددة

المحفظة تدعم ثلاث عملات:
- **YER** — الريال اليمني (`balance_yer`)
- **SAR** — الريال السعودي (`balance_sar`)
- **USD** — الدولار الأمريكي (`balance_usd`)

### 2.3 صلاحيات RLS (Row Level Security)

بعد إصلاح الـ recursion في migration 017:

```sql
-- دالة SECURITY DEFINER تتحقق من صلاحية الأدمن دون recursion
CREATE FUNCTION public.is_admin() RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.users
    WHERE id = auth.uid() AND role IN ('admin', 'owner', 'super_admin')
  );
$$ LANGUAGE sql SECURITY DEFINER;

-- السياسات:
-- 1. قراءة عامة على جداول الكتالوج (sections, sub_sections, api_games, ...)
-- 2. وصول كامل للأدمن على كل الجداول (عبر is_admin())
-- 3. المستخدم يقرأ/يعدّل صفه فقط في users
-- 4. المستخدم يقرأ طلباته ومعاملاته فقط
```

## 3. تكامل G2Bulk — التفاصيل الكاملة

### 3.1 تدفق شراء اللعبة الكامل (مثل G2Bulk)

```
المستخدم يختار لعبة (PUBG مثلاً)
        │
        ▼
[1] جلب الحقول المطلوبة للعبة
    POST /v1/games/fields  { game: "pubgm" }
    الرد: { fields: ["userid"] }
        │
        ▼
[2] المستخدم يدخل الـ ID
        │
        ▼
[3] التحقق من ID اللاعب وإظهار اسمه
    POST /v1/games/checkPlayerId  { game: "pubgm", user_id: "5679523421" }
    الرد: { valid: "valid", name: "PRIME・AKASH", openid: "4158..." }
        │
        ▼
[4] عرض اسم الحساب للمستخدم قبل تأكيد الشراء
    "الاسم: PRIME・AKASH"
    "ID: 5679523421"
        │
        ▼
[5] المستخدم يختار باقة (مثلاً "60 UC — $0.99")
    GET /v1/games/pubgm/catalogue
        │
        ▼
[6] تنفيذ الشحن
    POST /v1/games/pubgm/order  { catalogue_name: "60 UC", player_id: "..." }
    الرد: { success: true, order: { order_id: 12345, status: "PENDING" } }
        │
        ▼
[7] خصم الرصيد من محفظة المستخدم
    supabaseService.updateBalance(userId, 'USD', price, 'subtract')
        │
        ▼
[8] Polling لحالة الطلب كل 10 ثوانٍ (حتى 12 محاولة)
    POST /v1/games/order/status  { order_id: 12345, game: "pubgm" }
        │
        ├─── COMPLETED → عرض عناصر التسليم (أكواد، أرقام تسلسلية)
        ├─── FAILED → استرداد الرصيد تلقائياً + تحديث الطلب إلى 'refunded'
        └─── PENDING → متابعة الانتظار
```

### 3.2 بنية الأقسام بعد الإصلاح

**قبل الإصلاح (مكسور):** كل فئة من G2Bulk كانت تُنشئ قسماً رئيسياً جديداً، فتنتهي الشاشة الرئيسية بعشرات الأقسام.

**بعد الإصلاح (بنية G2Bulk):**

```
الصفحة الرئيسية
├── الأقسام العادية (محلي، اتصالات، كهرباء، ...)
├── G2Bulk (قسم رئيسي واحد — type='api')
│   ├── Airbnb US (sub-section)
│   ├── Amazon Japan (sub-section)
│   ├── Amazon Saudi Arabia (sub-section)
│   ├── Google Play Saudi Arabia (sub-section)
│   ├── PSN Qatar (sub-section)
│   ├── Apple iTunes Saudi Arabia (sub-section)
│   └── ... 151 sub-section
└── الألعاب (قسم رئيسي — type='games')
    └── 206 لعبة (من جدول api_games مباشرةً)
```

### 3.3 الأرقام الفعلية بعد المزامنة

| الم проз | العدد |
|---------|-------|
| ألعاب في `api_games` | 206 |
| فئات في `api_categories` | 151 |
| sub-sections تحت `g2bulk-root` | 151 |
| أقسام رئيسية مخصصة لـ G2Bulk | 2 (`g2bulk-root` + `g2bulk-games`) |

### 3.4 الأمان — Edge Function Proxy

**المشكلة الأمنية القديمة:** مفتاح G2Bulk API (`4882984fe50f...`) كان مكتوباً في كود العميل (`api-providers.ts`)، مما يعني أن أي شخص يستخرج APK يمكنه الحصول على المفتاح.

**الحل المُطبَّق:**

```
[تطبيق المستخدم/الأدمن]
        │
        │ POST /v1/games/checkPlayerId
        │ (بدون مفتاح G2Bulk — فقط anon key)
        ▼
[Supabase Edge Function: g2bulk-proxy]
        │
        │ يضيف X-API-Key: 4882984fe50f... (من secret)
        │ يمرر الطلب إلى G2Bulk
        ▼
[api.g2bulk.com]
        │
        │ يعيد الرد
        ▼
[g2bulk-proxy] يعيد الرد إلى التطبيق
```

- المفتاح موجود فقط كـ **Supabase secret** اسمه `G2BULK_API_KEY`
- التطبيقات تستدعي `https://kifmxseonkdsxuanznny.functions.supabase.co/g2bulk-proxy/...`
- الـ proxy يضيف المفتاح سرّاً ويمرر الطلب

**التحقق:** تم اختبار endpoint الـ proxy مباشرةً:
```
GET  /v1/games             → 206 لعبة ✓
POST /v1/games/checkPlayerId → { valid: "valid", name: "PRIME・AKASH" } ✓
```

## 5. ما زال ناقصاً (للإصلاح المستقبلي)

### 5.1 أولوية عالية

1. **إعادة كتابة `sub-sections-panel.tsx`** — لا يزال يستخدم Firebase path غير mapped. يحتاج إعادة كتابة بـ `supabaseAdmin` (نفس نمط `sections-panel.tsx`).
2. **إعادة كتابة `providers-panel.tsx`** — نفس المشكلة، للمزودين اليدويين (Yemen Mobile, YO, Sabafon, ...).
3. **إصلاح `manage-balance` edge function** — التحويل غير ذري (read-then-write بدون `FOR UPDATE`). يجب استخدام دالة `transfer_money()` PL/pgSQL الموجودة في schema.

### 5.2 أولوية متوسطة

4. **Webhook callback من G2Bulk** — `placeGameOrder` يدعم `callback_url` لكن لا يوجد edge function لاستقباله. الاعتماد الكامل على polling حالياً.
5. **تخزين صور الألعاب في Supabase Storage** — حالياً تُحمَّل من CDN مباشرةً. التخزين المحلي يعمل أوفلاين بعد أول مزامنة.
6. **إزالة مفتاح G2Bulk القديم من `api-providers.ts`** — الثابت `G2BULK_API_KEY` ما زال موجوداً كـ fallback. يجب إزالته بعد التأكد من أن كل الكود يستخدم الـ proxy.

### 5.3 أولوية منخفضة

7. **خريطة فئة → قسم محدد** — حالياً كل فئات G2Bulk تُنشأ تحت `g2bulk-root`. قد يريد الأدمن تخصيص فئة معينة لقسم آخر (مثلاً PSN إلى قسم "بطاقات هدايا").
8. **تتبع الـ stock** — عمود `product_packages.stock` موجود لكن `syncG2BulkProducts` لا يملؤه من API.
9. **سجل طلبات G2Bulk** — `getOrderHistory` function موجودة في `api-providers.ts` لكن لا توجد واجهة أدمن لعرضها.
10. **تخصيص سعر per-product** — جدول `price_overrides` موجود لكن لا توجد واجهة لاستخدامه.

## 7. كيفية الاستخدام

### 7.1 للمستخدم النهائي

1. يفتح تطبيق المحفظة
2. يضغط على قسم **"الألعاب"** في الصفحة الرئيسية
3. يختار لعبة (PUBG، Free Fire، MLBB، ...)
4. يُدخل معرف اللاعب (Player ID)
5. يضغط **"تحقق"** → يرى اسم حسابه
6. يختار باقة (60 UC، 120 UC، ...)
7. يضغط **"شراء"** → يتم خصم الرصيد وتنفيذ الشحن
8. يرى نتيجة الطلب (تم التسليم / قيد المعالجة / فشل مع استرداد تلقائي)

### 7.2 للأدمن

1. يفتح تطبيق الأدمن
2. يذهب إلى **"مزودي API"** → **"G2Bulk"**
3. في تبويب **"الإعدادات"**: يدخل مفتاح API (موجود مسبقاً)، يضغط "اختبار الاتصال"
4. في تبويب **"المزامنة"**: يضغط "مزامنة شاملة" → تتم مزامنة 206 لعبة + 151 فئة + 151 sub-section
5. في تبويب **"الألعاب"**: يتصفح الألعاب المُزامَنة بصورها
6. في تبويب **"الاختبار"**: يختار لعبة، يُجرب `checkPlayerId` و `placeOrder` قبل الإطلاق

### 7.3 للمطور

انظر **"دليل المطور"** المنفصل لتفاصيل:
- بنية المشروع
- كيفية إضافة مزود API جديد
- كيفية إضافة endpoint جديد لـ G2Bulk
- كيفية إنشاء migration جديد
- كيفية نشر edge function
